System Design

How E-Kart is structured — MVVM layers, Phase 1 mock boundaries, and the Phase 2 split between NestJS, FastAPI, and MongoDB.

MVVM + Clean Architecture Expo SDK 54 Phase 1 complete

Overview

E-Kart is a cross-platform Filipino grocery app. Phase 1 ships a complete frontend with mock services; Phase 2 replaces mocks with real APIs without rewriting ViewModels.

Phase 1 — Complete Frontend + mocks

Expo app, Zustand stores, mock services, AsyncStorage, file-system image URIs.

Phase 2 — Planned Backend integration

NestJS app API, FastAPI catalog search, MongoDB, object storage for captures.

MVVM + Clean Architecture

View — app/ routes + components/ (UI only)
↓ events / state
ViewModel — viewmodels/ (screen logic, no JSX)
↓ read / write
Store — stores/ (Zustand global state)
↓ IO
Service — services/mock · services/api
↓
Model — models/ (domain types, business rules)
  • Views never fetch directly All IO goes through ViewModels and services.
  • ViewModels stay UI-free No React Native component imports in viewmodels/.
  • Shared domain types Checklist, ForumPost, and related types live in models/.
  • Camera and OCR in services ViewModels orchestrate capture; services own IO boundaries.

Auth & access model

  • Phase 1 — mock session Local login sets session; single-user demo is sufficient.
  • Phase 2 — JWT auth NestJS handles credentials stored in MongoDB.
  • Checklist ownership Owned by user; share grants read (or edit — TBD) to another user.
  • Forum alias Public alias on posts; email and phone stay private on profile.
