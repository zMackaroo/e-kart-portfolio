Architecture · Technical Design

System Design

How E-Kart is structured — MVVM on the Expo client, NestJS API on MongoDB, and feature-flagged integration between mock and live services.

MVVM + Clean Architecture Expo SDK 54 + NestJS 11 Phase 1 + 2 complete

Overview

E-Kart is a cross-platform Filipino grocery app. Phase 1 shipped the Expo frontend; Phase 2 added e-kart-service (NestJS + MongoDB) and wired the app via typed API clients and USE_API.* feature flags.

Phase 1 — Complete Expo frontend

MVVM, Zustand stores, mock services for catalog/OCR, Gluestack UI, i18n.

Phase 2 — Complete NestJS + MongoDB

JWT auth, checklists, forum, AI generate, analytics, promos; app integration (spec 20).

MVVM + Clean Architecture

View — app/ routes + components/ (UI only)
↓ events / state
ViewModel — viewmodels/ (screen logic, no JSX)
↓ read / write
Store — stores/ (Zustand global state)
↓ IO
Service — services/mock · services/api
↓
Model — models/ (domain types, business rules)
  • Views never fetch directly All IO goes through ViewModels and services.
  • ViewModels stay UI-free No React Native component imports in viewmodels/.
  • Shared domain types Checklist, ForumPost, and related types live in models/.
  • Camera and OCR in services ViewModels orchestrate capture; services own IO boundaries.

Auth & access model

  • Phase 1 — mock session Local login sets session; single-user demo is sufficient.
  • Phase 2 — JWT auth (live) Register, login, session restore; Bearer token on API calls; 401 clears session.
  • Checklist ownership Owned by user; share grants read (or edit — TBD) to another user.
  • Forum alias Public alias on posts; email and phone stay private on profile.
