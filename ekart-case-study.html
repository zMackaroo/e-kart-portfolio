Product Design · Mobile Development · Case Study

E-Kart

A Filipino grocery app that helps households plan smarter palengke runs — with budget-aware checklists, multi-store price comparison, community tips, and AI-generated meal plans.

Role: Product & Engineering Platform: iOS & Android Stack: Expo · NestJS · MongoDB Phase: 1 + 2 complete (20 specs)

Overview

Filipino grocery shopping is rarely a single-store, single-list errand. Families juggle wet markets, supermarkets, budgets, and meal planning — often across multiple apps or handwritten lists. E-Kart consolidates that workflow into one mobile experience built specifically for the Philippine context.

Phase 1 delivered a fully navigable Expo frontend with mock services. Phase 2 added e-kart-service — a NestJS API on MongoDB — and wired the app end-to-end via JWT auth, typed API clients, and Zustand store hydration. Catalog search and OCR remain mock-only until dedicated backends exist.

Problem

For shoppers

Lists live on paper or chat threads. Prices vary by store with no easy comparison. Meal planning and grocery buying are disconnected, leading to repeat trips and budget surprises.

For the product

Generic grocery templates ignore Filipino categories (ulam, palengke staples) and local store names. A sustainable architecture must support OCR, AI generation, and community content without becoming a monolith.

Goals

  • Plan with confidence Master checklists with estimated vs actual spend, swipe actions, and share.
  • Compare before you buy Search catalog items across SM, Puregold, Robinsons, and more.
  • Learn from the community Reddit-style forum with votes, subforums, media, and alias-based identity.
  • Generate Filipino meal plans AI-assisted grocery lists by category, frequency, meat type, and custom notes.

Solution

E-Kart is organized around five tabs — Home, Checklist, Catalog, Forum, and Profile — with settings reachable from profile. Each major flow follows MVVM: thin Expo Router screens, ViewModels for logic, Zustand stores for state, and mock services that mirror future API contracts.

  • Home Monthly spend snapshot, active checklist shortcut, analytics cards, store promos.
  • Checklist Create lists, capture items with camera + mock OCR, track budget, swipe edit/delete.
  • Catalog Search-driven price rows per store with PHP formatting.
  • Forum Feed filters, search, post detail with media, subforum drawer, create post flow.
  • Profile & Settings Alias, activity stats, forum history, account forms, English/Filipino, light/dark/system theme.

Featured flow: AI Generate Checklist

The most differentiated Phase 1 feature. Instead of starting from a blank list, users pick a meal category (Ulam, Kids Meal, Healthy Snacks, Desserts, Vegan), a planning window (3 or 7 days), optional meat types, and free-text notes (e.g. allergies, air-fryer preferences).

1. Configure Category chips, frequency, multiselect meat types, additional details textarea.
2. Generate Lottie loading screen with cycling status copy; backend calls NestJS generate endpoint (mock/OpenAI/Anthropic).
3. Preview Dishes grouped by day with ingredient lists, then a consolidated shopping list with grand total.
4. Save or discard Nothing persists until the user confirms — draft state lives in a dedicated store.

“No sauce, anything that can be air fried.” — User notes flow into the preview and will inform the Phase 2 LLM prompt. Phase 1 uses curated Filipino ingredient pools per category and meat type.

Design intent: local context beats generic AI grocery lists

Process

Development was spec-driven: 11 numbered frontend specs (setup through AI generate), each with routes, ViewModels, acceptance criteria, and progress tracked in-repo. This kept scope visible and made it easy to extend Phase 1 (Spec 11) without renumbering backend work (Specs 12–14).

Research & context

Product overview, navigation map, data models, and optional grocery UI kit references for consumer patterns — adapted for Filipino workflows, not copied screen-for-screen.

Build & iterate

MVVM folders from day one, Gluestack design tokens, i18n shell, mock services swappable for APIs. Legacy template code removed once E-Kart screens replaced stubs.

Technical architecture

Expo SDK 54, Expo Router file-based navigation, Gluestack UI, Zustand with AsyncStorage persistence for auth/settings/profile, and TypeScript throughout. ViewModels expose screen state; services own IO; models stay framework-agnostic.

Expo SDK 54 React Native TypeScript NestJS 11 MongoDB JWT Expo Router Gluestack UI Zustand OpenAPI MVVM

Key decisions

  • API-first integration USE_API.* flags in constants/config.ts swap mock services for NestJS clients per domain — ViewModels unchanged.
  • Preview before persist (AI flow) Generated checklists are drafts until explicitly saved — reduces bad-list anxiety and matches real AI UX.
  • Alias-based forum identity Privacy-friendly display names synced to forum posts when profile alias edits.
  • PHP + Filipino copy Currency formatting and bilingual strings (English / Filipino) from Spec 02 onward.

Phase 2: Full-stack integration

Specs 13–20 built e-kart-service and connected the Expo app. The API runs at /api/v1 with Swagger docs, JWT auth, and MongoDB persistence.

  • Auth & profile (specs 14–15) Register, login, session restore, alias/email/phone/password, preferences sync.
  • Checklists (spec 16) CRUD, items, share by email, presigned S3 upload URLs for item photos.
  • AI generate (spec 17) POST /checklists/generate — Zod validation, rate limit, mock/OpenAI/Anthropic providers.
  • Forum (spec 18) Subforums, posts, comments, votes; seeded starter content on first boot.
  • Analytics & promos (spec 19) Monthly spend snapshot and home footer promos from MongoDB-backed API.
  • App integration (spec 20) Register screen, pull-to-refresh on home/checklist/forum, logout clears stores, full-stack dev docs.

Results

End-to-end smoke path works locally: register → create checklist → generate AI list → forum vote/comment → profile alias edit → home analytics refresh. TypeScript and ESLint pass on both repos; backend e2e suite covers API contracts.
